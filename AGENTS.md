# Repository Guidelines

## Project Structure & Module Organization
Core scraper logic lives in `src/`. `google_maps_scraper.py` orchestrates Playwright flows, while subpackages handle configuration (`config/`), domain models (`models/`), navigation helpers (`navigation/`), page extraction (`scraper/`), persistence (`persistence/`), and shared utilities (`utils/`). CLI entry points (`main.py`, `main_new.py`) sit at the root with `config.yaml`, which defines browser, grid, and output defaults. The Flask dashboard and API operate from `web/`—`app.py`, templates, and static assets—reusing the same config. Root-level `test_*.py` scripts cover scraper modes and API integration, and CSV/log artefacts generated by runs stay alongside the scripts for quick review.

## Build, Test, and Development Commands
- `python -m venv venv && source venv/bin/activate` (or `.\venv\Scripts\activate`) to create an isolated environment.
- `pip install -r requirements.txt` for scraper dependencies; add `-r web/requirements_web.txt` before running the dashboard.
- `python main.py -s "coffee shops" -t 20` executes the classic CLI scraper; `python main_new.py --scraping-mode coverage` enables grid-balanced runs.
- `python web/app.py` starts the Flask interface at http://localhost:5000.
- `pytest` runs automated checks; append `--cov=src` for coverage snapshots.
- `black src web tests *.py` and `flake8 src web` enforce formatting and linting.

## Coding Style & Naming Conventions
Target Python 3.9 compatibility with Black defaults (4-space indent, 88-char lines). Prefer snake_case for modules and functions, PascalCase for classes, and descriptive logger names. Keep type hints and docstrings in line with patterns in `src/google_maps_scraper.py`. YAML keys in `config.yaml` remain lowercase with underscores; avoid introducing mixed-case variants.

## Testing Guidelines
Add new tests to files named `test_*.py`. Unit tests should mock Playwright calls, while integration checks can mirror `test_integration.py` but must note prerequisites such as a running `python web/app.py`. Aim for >80% coverage on modified modules and verify with `pytest --cov=src --cov-report=term-missing`.

## Commit & Pull Request Guidelines
Commits use short, imperative subjects (“Add web interface”, “Refactor review extraction”) matching existing history. Group related changes per commit and expand in the body when altering scraping behavior or configuration schemas. Pull requests should summarise scope, list validation steps (`pytest`, sample scrape command, dashboard smoke test), link issues, and attach screenshots or CSV snippets when UX or output changes.

## Configuration & Security Tips
Treat `config.yaml` as environment-specific: adjust Chrome paths, bounds, and review limits locally, and avoid committing credentials. Prefer `.env` or per-host overrides ignored by git. When deploying `web/app.py`, set a unique `SECRET_KEY`, serve through HTTPS, and prune generated CSVs/logs before sharing to prevent leaking customer data.
